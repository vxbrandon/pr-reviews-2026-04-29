### 3.5 Class D — Pipeline / service flows (the meat)

8 new + 6 modified pipeline branches. Highlights:

- **P-CAL** — per-match calibration sweep. `MatchCalibrationService.calibrate_match()` reads `live_odds:{gid}`, computes divergence vs market, optimises home/away/player/team factors, writes `match_cal_adj:{gid}`.
- **P-SIM-CAL** — calibrated match simulation. Distinct cache namespace `odds:sim:calibrated:{gid}:{actions}:adj={fingerprint}`. Replaces post-hoc lambda multiplication with calibrated marginals fed into the vine copula.
- **P-CLUSTER** (W5) — `PlayerClusterFitter` produces per-player-cluster params; `PlayerClusterPredictor` reads them at predict-time; blend with zonal model in `_sample_player_counts`.
- **P-REFIT** — blue-green pipeline refit. `RefitManager.start_refit()` runs validation (`predict_match` smoke on 3 recent games, `home/away ∈ [0.1, 5.0]`, `n_teams ≥ 4`) before the atomic swap.
- **P-ZAP / P-ZAP-BATCH / P-PARLAY** — modified to use `PricingContext` (snapshots adj once per request) + `*_fingerprint` (mixes only the adj slice each cache reads). Closes the URA-T17/T19 staleness class.

Full per-pipeline walks in [`_pr-140-flows-pipelines.md`](_pr-140-flows-pipelines.md). Cross-class end-to-end walks (parlay, league migration, ingest→live odds, lineup scrape) in [`_pr-140-flows.md`](_pr-140-flows.md) §"Cross-class flow walks".

### 4.5 Notable architectural decisions

1. **Six-folder service taxonomy** — flat → grouped, history preserved via `git mv`.
2. **Two alembic histories** — both must be run on deploy. `alembic_soccer_data.ini` is deleted.
3. **New `ext_odds` Postgres schema** — 23 tables in a dedicated schema, fed by sister project `src/ext_odds_streaming/`. Not a refactor; a new persistence subsystem.
4. **Source-aware data plane** — `events`/`match_statistics`/`player_statistics`/`lineups` widen PKs/UNIQUEs to include `source`. Every consumer that did `scalar_one_or_none()` on `(game_id, player_id)` is at risk; `lineup_source_priority.py` exists specifically to deduplicate.
5. **Calibration as a peer subsystem** — `MatchCalibrationService` elevates ad-hoc factor adjustments to a layered service with its own enable flag, cooldowns, divergence threshold, extreme-policy alerting (clamp / passthrough / passthrough_with_alert).
6. **Vig math centralised** — `vig_policy.py` deduplicates 3 prior copies.
7. **W1 telemetry pipeline** — `RequestAnalyticsService` is the first time a service in `application/` runs an in-process consumer thread. Worth flagging as a layering departure (existing Kafka consumers live in `infrastructure/kafka/`).
8. **Strict alembic autogenerate** — expect noisier autogenerate output post-merge; review carefully.
9. **CI footprint shrunk** — `.github/workflows/claude-code-review.yml` removed (−78). Whatever automated review existed there is no longer running.

### 6.5 The CI gap (single most important fix-before-merge in this section)

The 73 new unit tests under `tests/unit/` are **not invoked by any workflow on PR-to-`dev`**. Only `test-pipeline.yml` (WebSocket Docker smoke) and `flow-tests.yml` (`tests/flow_tests/` only) run. There is no coverage threshold and no dedicated `pytest tests/unit` job.

**What this means**: `make check` works locally and the 73 tests pass — but CI signals "green" without ever running them. Any subsequent regression to W2/calibration/clusters/etc. could merge without breaking the PR check.

**Fix**: add a `unit-tests.yml` workflow that runs `pytest tests/unit/ -v --cov=src/backend_odds --cov-fail-under=<threshold>` on PR-to-`dev`.
